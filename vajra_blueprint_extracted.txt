2. Executive Summary
Vajra is a recipient-authenticated payment handshake for Monad. A recipient creates exact payment terms, 
authenticates those terms with a registered device passkey or wallet signature, and shares a link or QR code. A 
sender opens the request, verifies the signed terms without pasting an address, pays the exact native-MON amount 
once, and receives a permanent settlement receipt after the transaction is finalized.
ONE SENTENCE  Vajra lets the recipient authenticate the exact amount, recipient, expiry and optional payer restriction 
before the sender pays, preventing silent tampering and duplicate settlement.
Why this product exists
• Wallet-address copy/paste is error-prone and vulnerable to social engineering, address poisoning and complete-
link substitution.
• Traditional payment links often treat amount and recipient metadata as editable convenience fields rather than 
recipient-authenticated terms.
• Users struggle to distinguish wallet signature, transaction broadcast, block inclusion and irreversible finality.
• Passkeys can provide a more human request-creation experience without granting authority to spend the recipient 
wallet balance.
What makes Vajra technically distinctive
Capability Product effect
WebAuthn/P-256 request authentication Recipient can authorize a payment request with Face ID, Touch 
ID, Windows Hello, device PIN or a security key after one-time 
registration.
Exact single-use settlement The first valid payment succeeds; a second attempt fails at 
contract level.
Domain-separated request digest The same request cannot be replayed on another chain or 
another Vajra deployment.
Visible Monad finality The receipt distinguishes wallet approval, broadcast, inclusion and 
finalization using observed chain state.
Non-custodial atomic transfer Vajra does not maintain user balances; successful payment is 
transferred directly to the signed recipient.
No privileged administrator No owner can redirect, seize, pause or upgrade payment logic.
Primary judge demo
1. Recipient registers a device passkey with a Monad wallet.
2. Recipient creates a 0.01 MON request and authenticates it with the device passkey.
3. Sender opens the link and verifies the exact terms before connecting a wallet.
4. Sender pays and watches the real transaction move from broadcast to finality.
5. The receipt becomes permanently sealed from real event and state data.
6. A modified amount fails signature verification.
7. A second payment attempt fails onchain because the request is already consumed.

Product principles
• One real action over many fake features: A single secure payment handshake must be complete end to end.
• No pasted destination: The sender must not be required to paste a recipient address into the payment flow.
• No silent mutation: Changing any authenticated field invalidates the request.
• No fake success: Success requires a successful receipt, matching event and paid onchain state.
• No hidden custody: The contract must not expose user deposit balances or administrative withdrawal paths.
• No fake identity claims: A signature proves wallet or credential control, not the real-world identity of a person.
• No chain decoration: Monad-specific capabilities must materially improve the product experience.
Product name and language
The product name is Vajra. Feature labels may use functional terms such as Vajra Request, Vajra Touch, Vajra 
Code and Vajra Seal. These names are product vocabulary, not a visual-design decision.

4. Problem Definition and Success Metrics
User problem
A sender who receives a wallet address and amount through chat must trust that the destination, amount and 
network were copied correctly and not substituted. Even after broadcasting, the sender may not know whether the 
payment merely entered a mempool, was included in a block or became final. The recipient must repeatedly open a 
wallet and sign messages or transactions to create trustworthy requests.
Target outcome
A recipient should be able to create a request in seconds using a device credential, while a sender should be able to 
verify every material term before connecting a wallet and complete a one-time payment with a verifiable final receipt.

7. User Journeys
Journey A - First-time recipient passkey setup
1. Connect recipient wallet on the configured Monad network.
2. Read a plain-language explanation: the passkey may authorize incoming payment requests but cannot spend 
wallet funds.
3. Create a WebAuthn credential with user verification required.
4. Locally verify the credential response and public key.
5. Submit a wallet-signed registration transaction containing the P-256 public key, credential identifier hash, relying-
party identifier hash and initial key version.
6. Wait for successful transaction receipt and matching registration event.
7. Re-read the registry and show the passkey as active only after onchain confirmation.
Journey B - Recipient creates a request
1. Enter the exact MON amount, expiry, short memo and optional exact payer.
2. Read the latest chain timestamp and produce a bounded issuedAt / expiresAt window.
3. Generate a cryptographically random 32-byte nonce.
4. Construct the canonical request digest using the configured chain and contract domain.
5. Use WebAuthn to sign the digest as the challenge, or explicitly select wallet-signature fallback.
6. Verify the signature locally before generating the share artifact.
7. Create a versioned URL-fragment payload and QR code.
8. Display the full request terms, full recipient address, short Vajra Code and expiry.
Journey C - Sender verifies and pays
1. Open the request without connecting a wallet.
2. Strictly decode and validate the payload.
3. Verify chain ID, canonical contract, request digest and authentication mode.
4. Call the contract inspection function with a prospective payer address when known.
5. Display full recipient address, amount, expiry, memo, authentication method, passkey version and Vajra Code.
6. Connect a wallet only after the request is valid.
7. Switch to the configured Monad network if required.
8. Re-inspect and simulate immediately before payment.
9. Submit exact msg.value through fulfill.
10. Track transaction hash, receipt, event and finality.
11. Render a sealed receipt only after all success checks pass.
Journey D - Recipient revokes an unpaid request
1. Open a previously created request from local history or its share link.
2. Connect the recipient wallet.
3. Submit revoke with the exact request data.
4. Show that revocation remains pending until confirmed.
5. After confirmation, re-read request state and prevent payment.
Journey E - Lost device or passkey rotation
1. Connect the recipient wallet.
2. Register a replacement passkey and increment key version, or deactivate the active passkey.

8. Product Requirements - Functional
Request creation
FR-001 [MUST] The application MUST construct a canonical request containing recipient, optional payer, exact amount, 
issuedAt, expiresAt, random nonce, memoHash, authentication mode and authentication version.
FR-002 [MUST] The amount MUST be a positive integer number of wei and MUST NOT use JavaScript floating-point 
arithmetic.
FR-003 [MUST] The application MUST obtain issuedAt from recent chain state rather than relying only on the device clock.
FR-004 [MUST] The request lifetime MUST be at least 60 seconds and at most 30 days in v1.
FR-005 [MUST] The memo MUST be limited to 96 UTF-8 bytes after normalization.
FR-006 [MUST] The nonce MUST be 32 cryptographically random bytes and MUST NOT be derived from timestamp, 
amount or address.
FR-007 [MUST] The application MUST verify the generated request locally before it can be shared.
FR-008 [MUST] Request creation after passkey registration MUST require no transaction and no gas.
Authentication
FR-009 [MUST] Passkey mode MUST use a registered P-256 public key, WebAuthn user verification, the canonical 
request digest as challenge and the active key version.
FR-010 [MUST] Wallet fallback MUST use EIP-712 domain separation and MUST support ERC-1271 through 
SignatureChecker or an equivalent audited implementation.
FR-011 [MUST] The passkey MUST NOT authorize any function that transfers funds from the recipient wallet.
FR-012 [MUST] Rotating or deactivating a passkey MUST invalidate unpaid requests signed by earlier versions.
FR-013 [MUST] The frontend MUST label passkey authentication as registered-device authentication, not verified person 
or verified identity.
Sharing and verification
FR-014 [MUST] The request payload MUST be carried in the URL fragment rather than a query string.
FR-015 [MUST] The decoder MUST reject unknown versions, unknown keys, oversized payloads, malformed numbers, 
unsafe integers and invalid signatures.
FR-016 [MUST] The payment page MUST be readable and verifiable without wallet connection.
FR-017 [MUST] The payment page MUST display the full recipient address, exact amount, expiry, memo, payer restriction, 
chain, contract and request code.
FR-018 [MUST] The frontend MUST obtain chain name, explorer and contract metadata from trusted configuration, never 
from the shared payload.
FR-019 [MUST] Mutating any authenticated field MUST invalidate the request and remove the payment action.
Settlement
FR-020 [MUST] The sender MUST pay the exact signed amount using msg.value; underpayment and overpayment MUST 
revert.
FR-021 [MUST] A payer-restricted request MUST only be fulfillable by the exact signed payer address.
FR-022 [MUST] A request MUST settle at most once.
FR-023 [MUST] A revoked, expired, invalid, old-key-version or already-paid request MUST NOT display an enabled 
payment action.
FR-024 [MUST] The contract MUST update state before making the external native-MON transfer.
FR-025 [MUST] A failed recipient transfer MUST revert the entire transaction and leave the request unused.
FR-026 [MUST] The payment contract MUST NOT maintain user deposit balances.
Receipt and finality
FR-027 [MUST] The frontend MUST retain the transaction hash after broadcast even if the wallet disconnects or account 
changes.
FR-028 [MUST] Success MUST require receipt status success, a matching settlement event and a paid onchain state.
FR-029 [MUST] The finality seal MUST use observed chain state and MUST NOT display a hardcoded elapsed time.
FR-030 [MUST] When the finalized block tag or preferred finality mechanism is unavailable, the UI MUST clearly label the 
fallback confirmation method.

• No protocol fee.
• No pause switch.

• No delegatecall.

12. Passkey and Signature Architecture
Security boundary
PASSKEY AUTHORITY  A passkey authorizes only an incoming payment request whose recipient is the wallet that 
registered the key. It cannot spend the recipient wallet balance, transfer tokens, change the payout address or manage 
wallet permissions.
WebAuthn proof requirements
• The authenticator data relying-party identifier hash must match the registered production RP ID hash.
• The user-presence bit must be set.
• The user-verification bit must be required for production requests.
• The client data type must be webauthn.get.
• The challenge must encode the canonical request digest exactly.
• The P-256 signature must verify against the active registered public key.
• The request authVersion must equal the active passkey version.
• The verifier must reject malformed offsets, malformed JSON, truncated authenticator data and oversized 
signatures.
Registration
Passkey registration is authorized by the recipient wallet. The contract may accept the selected public key without 
requiring an onchain attestation from the new key because the wallet transaction itself authorizes the binding. The 
frontend must still verify the WebAuthn registration response locally to prevent accidental registration of malformed 
data.
Rotation and deactivation
Every successful rotation or deactivation increments the credential version. Existing unpaid passkey requests carry 
the prior version and become invalid without requiring a per-request revocation transaction.
Wallet fallback
Wallet mode uses the same request digest and EIP-712 domain. SignatureChecker or an equivalent audited 
implementation should support EOAs and ERC-1271 smart wallets. Signature validity must be checked at fulfillment 
time and must never be cached as permanent truth.
Origin and RP-ID controls
• Production passkeys should use a stable application RP ID such as app.vajra.xyz or vajra.xyz after domain 
ownership is confirmed.
• Local development must use a separate environment and must never silently reuse production registration data.
• The contract registry should store rpIdHash per credential and compare it with authenticator data.
• The application must deploy HSTS, secure DNS and strict origin controls because passkey safety depends partly 
on domain control.

13. Settlement Logic and Contract Safety
Fulfillment order
1. Validate static request fields and bounded time window.
2. Compute the canonical request ID.
3. Require request status Unused.
4. Require current block time not later than expiresAt.
5. Require authVersion and passkey state to be current when passkey mode is used.
6. Require msg.sender to equal the restricted payer when payer is non-zero.
7. Verify wallet or passkey authentication.
8. Require msg.value to equal amount exactly.
9. Set request status to Paid.
10. Write the settlement record.
11. Transfer native MON to the recipient with a checked low-level call.
12. Emit exactly one PaymentFulfilled event.
Checks-effects-interactions
statusOf[id] = RequestStatus.Paid;
settlementOf[id] = settlement;
(bool ok, ) = payable(request.recipient).call{value: request.amount}("");
if (!ok) revert NativeTransferFailed();
emit PaymentFulfilled(...);
Because a revert rolls back all state changes, a rejected native transfer leaves the request unused. fulfill must also 
be protected by nonReentrant.
Direct transfers
receive() external payable { revert DirectPaymentsDisabled(); }
fallback() external payable { revert DirectPaymentsDisabled(); }
Forced native transfers may still be possible at the EVM level. The contract must not expose a privileged sweep 
function merely to clean such funds. Documentation must state that Vajra does not intentionally hold user balances.
Revocation race
Payment and revocation may compete. Whichever transaction finalizes first determines the terminal state. The UI 
must not describe a pending revocation as effective.

Receipt verification
The receipt layer must independently verify all available evidence before showing the final seal.
1. Transaction receipt status is success.
2. PaymentFulfilled event is emitted by the canonical Vajra contract.
3. Event requestId equals the locally computed request ID.
4. Event payer, recipient, amount, auth mode and version match the request and connected payer.
5. statusOf(requestId) returns Paid.
6. settlementOf(requestId) matches the event.
7. The transaction block is finalized according to the configured finality source.
Finality tracking
• Preferred path: query the finalized block tag and compare its height with the transaction block.
• Fallback path: use a documented confirmation strategy when the finalized tag is unavailable.
• Optional advanced demo path: use Monad Execution Events when dedicated node infrastructure is available.
• Never display a hardcoded 800 ms claim as the observed transaction time.
• If the RPC becomes unavailable after broadcast, retain the hash and present an honest verification-pending state.

21. Security Invariants
The following properties are release blockers. Invariant tests should exercise randomized action sequences and 
malicious recipients.
1. A request can settle at most once.
2. A paid request can never become revoked or unused.
3. A revoked request can never become paid.
4. Successful settlement transfers exactly the signed amount.
5. Successful settlement transfers only to the signed recipient.
6. A restricted request can only be settled by the signed payer.
7. An expired request never transfers MON.
8. An invalid signature or WebAuthn proof never transfers MON.
9. A passkey request with an inactive or stale key version never transfers MON.
10. A failed recipient transfer leaves the request unused.
11. Reentrant execution cannot create a second settlement or corrupt state.
12. No privileged account can change a request, redirect a payment or recover funds.
13. No function can execute arbitrary user-supplied calldata.
14. The TypeScript request ID always equals the Solidity request ID for the same canonical fields.
15. Every successful fulfillment emits exactly one matching PaymentFulfilled event.
16. The frontend never renders Paid before independent receipt, event and state checks succeed.

29. Residual Risks and Honest Limitations
1. A scammer can create a valid request for their own wallet. Vajra verifies control and integrity, not morality or legal 
identity.
2. A compromised recipient wallet can register an attacker-controlled passkey.
3. A compromised sender wallet or malicious browser extension can approve an unwanted transaction.
4. A phishing clone can imitate the frontend; users must verify the canonical domain and contract.
5. The public blockchain exposes sender, recipient, amount and timing.
6. Anyone with the share link can read its contents.
7. RPC providers may fail, lag or disagree temporarily.
8. Smart-wallet signature validity may change according to that wallet contract.
9. Recipient contracts may reject native MON, causing fulfillment to revert.
10. No test suite or audit can prove the complete absence of unknown vulnerabilities.
SECURITY LANGUAGE  Never market Vajra as loophole-free, impossible to hack or guaranteed safe. Correct 
language is: minimized attack surface, explicit security boundaries, tested invariants, immutable settlement logic and 
independently verifiable results.

Appendix B - Validation Codes
Code Meaning
Valid Request is currently payable by the prospective payer.
ZeroRecipient Recipient is the zero address.
ZeroAmount Amount is zero.
InvalidNonce Nonce is zero or malformed.
InvalidTimeWindow issuedAt/expiresAt violate bounds.
Expired Current block time is after expiresAt.
AlreadyPaid Request status is Paid.
Revoked Request status is Revoked.
WrongPayer Prospective payer does not match the restriction.
InactivePasskey Recipient has no active passkey for passkey mode.
WrongPasskeyVersion Request version does not equal active version.
InvalidWalletSignature EIP-712 / ERC-1271 signature failed.
InvalidPasskeyProof WebAuthn/P-256 proof failed.
IncorrectValue msg.value does not equal amount.
